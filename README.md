# Abstract-vs-Interface-Design-Debate

What worked better with interfaces was using them to give specific instructions to certain classes. For example, some classes were described while others were not. This showed the versatility of interfaces and how they are not as restricted as abstract classes are. This feature would come in handy when certain classes need specific instructions to operate, providing key functions to some classes while protecting the other classes from unnecessary changes.
What was more reusable was the abstract class was the draw, area, and name functions. Every class had either a name, area, or was drawn. Thus, it was used in the abstract method to reduce the redundancy of code. 
Interfaces would certainly scale better. Even though Abstract classes provide the foundation for multiple classes, they are not as versatile as interfaces. As described before, interfaces can be used to add instructions or details to other classes without affecting the other classes. They could also be used to connect other classes together without needing to have an abstract class connected to them. This allows greater flexibility in code that could not be used otherwise. 
I would choose the abstract method. The reason would be that you could replace multiple different interfaces with one abstract class. With what we have done so far, and with the examples we have done, there seemed to be lots of redundancy with the use of interfaces. Their only use as of late was to give specific instructions to certain classes. Now, I am well aware of the fact that interfaces could be used more thoroughly and have more uses than what we have done in class. The abstract class, from what I have seen, takes care of more functions efficiently than the interface based design.
